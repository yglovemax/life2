# Nexa 占卜 APP AI API 管理后台开发需求文档

## 1. 项目目标

本项目要开发一个用于 Nexa 占卜 APP 的 AI API 管理后台。

第一期目标不是开发一个完整大系统，而是做一个一周内可以实际使用的版本，用来解决当前 AI 内容调试、字段协作、接口排查和版本发布困难的问题。

目前 Nexa 用户端页面已经可以显示，主要问题是 AI 返回内容仍然需要反复调试，且排查链路很长。产品经理发现内容问题后，需要同时协调前端、App 后端、大模型 API 后端、Prompt 调试人员和命理/占星内容负责人，效率很低。

第一期后台要成为一个轻量但可用的协同调试平台，让团队能看到：

1. 某个页面由哪些内容模块组成。
2. 每个模块当前使用什么 Prompt。
3. 每个模块调用什么大模型。
4. 每个模块依赖哪些算法数据和知识库内容。
5. 每个模块最终返回哪些 JSON 字段。
6. 某次请求输入给大模型的内容是什么。
7. 大模型原始返回了什么。
8. 最终返回给前端的 JSON 是什么。
9. 哪个模块有问题，由谁负责，处理到哪一步。

第一期先用星座/占星页面验证整套协同系统是否可行。系统稳定后，再把八字成果接入进来。

## 2. 当前开发策略

第一期采用低风险验证路径：

1. 八字继续按照原来的开发模式推进。
2. 八字模块、提示词、前端工作不中断。
3. 星座/占星页面与 AI API 管理后台同步开发。
4. 用星座/占星页面验证后台的模块管理、Prompt 管理、字段契约、测试、发布和追踪流程。
5. 等后台系统稳定、可用、协作流程跑通后，再逐步接入八字成果。

这样可以避免把已经做了一大半的八字工作切换到一个尚未验证的新系统里，造成进度卡住。

## 3. 一期开发范围

第一期只做以下两个用户端页面对应的 AI API 管理与调试能力：

1. 出生星盘解读页
   - 参考 UI：`App 用户端 UI/出生星盘解读页.png`
   - API 目标：为出生星盘解读页提供星盘事实数据、模块解读内容、整页 JSON 输出和调试追踪。

2. 每日星座运势页
   - 参考 UI：`App 用户端 UI/每日星座运势页.png`
   - API 目标：为每日星座运势页提供每日运势内容、模块解读内容、整页 JSON 输出和调试追踪。


## 4. 核心痛点

当前开发流程存在以下问题：
1. 前端将能读取的 JSON 字段发给后端程序员。
2. 后端程序员根据 JSON 样本制作大模型返回内容。
3. 产品经理发现内容不符合预期后，需要让前端/后端人员写更改请求。
4. API后端再根据新样本重新制作。
5. 前端，后端和API后端的程序员并不一定理解产品最终想要的内容。
6. 容易出现前端后段认为字段没问题、但实际样本有问题，输出有问题且不知道谁负责修改的责任争议
7. 一次小的内容调整，可能需要 24 小时以上才能完成一轮修改，甚至两周都无法修改
8. 如果问题来自大模型输出，还需要再找 Prompt 调试人员排查。
9. 如果问题来自算法数据，还需要找接口或算法负责人排查。
10. 如果问题来自知识库或占星规则，还需要找内容专家排查。
11. 如果问题来自字段，就需要两个后端的字段和前端都修改，如果是这样的情况，就会导致两周以上都无法修改

第一期后台要解决的核心问题是：

1. 让产品经理能直接看到字段含义和内容样例。
2. 让 Prompt 调试人员能直接看到最终请求内容。
3. 让程序员能看到字段来源、算法数据和接口返回。
4. 让团队能快速判断问题来自前端字段、后端拼装、算法数据、Prompt、模型还是知识库。
5. 让不影响前端结构的内容调整尽量在后台完成。

## 5. 后端整合目标

当前系统存在两个后端：

1. 网页 APP 后端
   - 负责用户、页面、业务接口、前端 JSON 返回等。

2. 大模型 API 后端
   - 负责 Prompt 拼装、大模型调用、AI 内容生成、模型配置等。

第一期需要把两个后端打通，形成统一的 AI API 调试链路。

目标不是强制程序员把所有代码合并成一个项目，而是业务上必须做到：

1. App 后台可以稳定调用 AI API 管理后台提供的内容输出能力。
2. AI API 管理后台可以接收 App 后台传来的用户资料、页面标识、模块标识、日期等必要信息。
3. AI API 管理后台可以调度算法数据、Prompt、知识库、大模型和 Fallback。
4. AI API 管理后台最终返回前端页面需要的整页 JSON 或模块 JSON。
5. 产品经理可以在后台看到一次请求从 App 后端到大模型再到最终 JSON 的完整链路。

具体采用代码合并、接口网关、服务整合还是统一日志与调用层，由程序员根据现有系统决定。

## 6. 一期页面与模块

第一期按页面模块接入，不做整页大 Prompt。

每个页面由多个模块组成，每个模块都可以单独管理 Prompt、模型、字段、测试、成本、Fallback 和发布状态。

### 6.1 出生星盘解读页

出生星盘解读页是用户个人星盘解读页面。

页面特点：

1. 基于用户出生资料生成。
2. 星盘事实数据相对稳定。
3. 用户首次生成后可以保存。
4. 如果用户修改出生资料，需要重新生成。
5. 适合用于验证“事实算法数据 + AI 解读 + JSON 拼装”的流程。

页面模块按现有前端 UI 和 JSON 样本接入，至少包括：

1. 用户基础星盘信息
2. 出生星盘图
3. 星盘表格数据
4. 星盘详解
5. 占星主导模式
6. 占星主导元素
7. 各宫位解读
8. 星座特征
9. 星座符号
10. 其他占星术
11. 占星肯定语

### 6.2 每日星座运势页

每日星座运势页是每日变化内容。

页面特点：

1. 按日期变化。
2. 需要支持实时生成。
3. 后期可以支持缓存或提前生成。
4. 适合用于验证“每日内容生成 + 多模块并行 + 成本控制 + Fallback”的流程。

页面模块按现有前端 UI 和 JSON 样本接入，至少包括：

1. 用户星座基础信息。
2. 日运势天气分值
3. 占星日运
4. 事业运势
5. 爱情运势
6. 健康运势
7. 家庭运势
8. 财富运势
9. 学习运势
10. 每日寄语
11. 今日宜忌
12. 今日塔罗
13. 塔罗问答
14. 生物节律
15. 月相历
16. 关键星象事件
17. 每日占星 Tips
18. 周运势
19. 月运势
20. 年运势

实际模块名称和字段以当前前端 JSON 样本为准。

## 7. 页面生成逻辑

### 7.1 出生星盘解读页

用户打开出生星盘解读页时，后台需要：

1. 获取用户出生资料。
2. 计算或读取用户星盘事实数据。
3. 根据页面模块拆分需要生成的内容。
4. 各模块分别调用自己的 Prompt、知识库和大模型。
5. 生成结果保存。
6. 最终拼装成前端需要的整页 JSON。

如果用户修改出生日期、出生时间、出生地点或其他影响星盘的数据，需要重新生成相关内容。

### 7.2 每日星座运势页

用户打开每日星座运势页时，后台需要：

1. 获取用户星座、日期和必要的个性化信息。
2. 获取当日星象或星座运势算法数据。
3. 按模块生成内容。
4. 多个模块可以并行调用大模型。
5. 不同模块可以使用不同模型。
6. 某个模块生成失败时，返回该模块 Fallback 内容。
7. 最终拼装成前端需要的整页 JSON。

每日总览类内容优先返回。

## 8. 平台核心原则

1. 页面按模块管理，不按整页大 Prompt 管理。
2. 算法数据一次计算，多个模块共享使用。
3. AI 文案按模块分开生成、分开调用大模型。
4. 不同模块可以调用不同大模型，用于控制成本和质量。
5. 多个模块可以并行生成，提高页面返回速度。
6. 每个模块都可以单独修改、测试、灰度、回滚。
7. 某个模块生成失败，不影响整页返回，失败模块使用自己的 Fallback 内容。
8. 最终由后台把各模块结果拼装成前端需要的整页 JSON。
9. 所有模块必须支持输出字段契约。
10. 后台操作界面要简单，方便非程序员使用。
11. 内容调整和字段样例调整应尽量在后台完成。
12. 每次 AI 调用都要能追踪输入内容、模型返回、最终 JSON、Prompt 版本、算法数据和负责人。
13. 必须重视网络安全和知识产权保护。
14. 第一版就要支持新增模块，新增模块后可以进入测试、灰度、发布和回滚流程。

## 9. 平台架构

第一期建议做成轻量版本，只保留最必要的后台模块。

### 9.1 模块中心

模块中心是后台首页。

用于查看所有页面和模块。

模块中心需要提供新增模块入口。

列表字段包括：

1. 模块名称。
2. 所属页面。
3. 当前负责人。
4. 当前模型。
5. 当前版本。
6. 当前状态。
7. 今日调用量。
8. 今日成本。
9. 错误率。
10. Fallback 触发次数。
11. 待处理问题数。
12. 最近更新时间。

需要支持按页面、负责人、状态、错误、成本异常、Fallback 异常和待处理问题筛选。

### 9.2 模块详情页

模块详情页是每个模块的主要工作台。

需要包含：

1. 模块基础信息。
2. Prompt 五段式。
3. 模型选择。
4. 知识库选择。
5. 算法数据预览。
6. 输出 JSON 与字段契约配置。
7. Fallback 配置。
8. 测试按钮。
9. 发布按钮。
10. 历史版本。
11. 最近调用记录。
12. 当前待处理问题。
13. 问题负责人和处理状态。

模块详情页必须能并排查看：

1. 输入给大模型的内容。
2. 大模型返回的原始内容。
3. 最终返回给前端的 JSON 内容。

### 9.3 测试中心

测试中心用于集中测试模块效果。

第一期需要支持：

1. 选择页面。
2. 选择模块。
3. 选择测试用户。
4. 选择日期。
5. 选择模型。
6. 单模块测试。
7. 多模块批量测试。
8. 查看输出结果。
9. 人工评分。
10. 复现线上问题。
11. 对比修复前后的结果。

### 9.4 知识库

知识库第一期要简单。

需要支持：

1. 上传 Markdown 文档。
2. 人工录入知识条目。
3. 简单编辑知识内容。
4. 给知识内容打标签。
5. 查看某次生成命中了哪些知识内容。

如果开发成本可控，可以做轻量 RAG。

### 9.5 成本中心

成本中心用于查看模型调用成本。

第一期需要支持：

1. 查看总成本。
2. 查看当日成本。
3. 查看近 7 天成本。
4. 查看本月成本。
5. 按页面查看成本。
6. 按模块查看成本。
7. 按模型查看成本。
8. 按测试和正式调用区分成本。
9. 查看 Prompt Caching 命中率。
10. 查看异常成本提醒。

成本中心需要同时支持总览和明细：

1. 总览用于让项目经理快速知道今天、一周、本月整体花了多少钱。
2. 明细用于定位具体是哪个页面、哪个模块、哪个模型导致成本升高。
3. 成本异常时，需要能快速追溯到具体模块和调用记录。

### 9.6 发布中心

发布中心用于管理模块版本。

第一期需要支持：

1. 草稿。
2. 待测试。
3. 测试通过。
4. 待审批。
5. 灰度中。
6. 已上线。
7. 已回滚。
8. 已停用。

每个模块可以独立发布、独立灰度、独立回滚。

## 10. Prompt 管理

每个模块的 Prompt 必须采用五段式结构：

1. 共享静态前缀。
2. 模块专属输出规则。
3. 用户算法数据。
4. 用户偏好及写作规则。
5. 最终请求预览。

### 10.1 共享静态前缀

用于管理 Nexa 全局内容风格、安全边界、品牌语气和禁止事项。

### 10.2 模块专属输出规则

用于管理某个模块自己的写作要求、字数、段落、字段、禁止内容和输出格式。

### 10.3 用户算法数据

由后端算法或现有接口生成。

例如出生星盘、太阳星座、月亮星座、上升星座、行星位置、宫位、相位、每日星象等。

这部分由程序员和算法负责人维护，后台需要能查看和选择，不要求非技术人员编辑核心算法代码。

### 10.4 用户偏好及写作规则

来自用户个性化设置或产品配置。

例如语气、信息密度、情绪温度、结构偏好、用户昵称、职业、个人详情等。

### 10.5 最终请求预览

后台需要展示最终发送给大模型的完整内容。

如果输出不好，团队需要能判断问题来自：

1. 共享静态前缀。
2. 模块专属输出规则。
3. 算法数据。
4. 用户偏好。
5. 知识库。
6. 模型本身。

## 11. 输出 JSON 与字段契约管理

后台必须支持可视化字段编辑器，并把前端、后端、产品共同依赖的 JSON 样本变成可管理的字段契约。

每个字段需要说明：

1. 字段名称。
2. 字段用途。
3. 前端展示位置。
4. 字段示例。
5. 字段来源。
6. 是否由大模型生成。
7. 是否必填。
8. 字段负责人。
9. 当前状态。
10. 修改记录。

字段来源可以包括：

1. 算法计算结果。
2. AI 生成内容。
3. 固定配置内容。
4. 知识库内容。
5. 用户偏好内容。
6. 图片、图标、跳转链接等前端展示字段。

重要原则：

1. 不是所有字段都需要大模型生成。
2. 大模型只负责需要自然语言生成的部分。
3. 最终 JSON 由后台统一拼装。
4. 前端只按照约定 JSON 渲染页面。
5. 产品经理需要能直接看到字段最终展示什么内容，而不是只看到技术字段名。
6. 字段含义、样例、内容要求和当前线上版本必须统一记录。

## 12. 算法数据管理

第一期不做复杂算法平台。

第一期只需要把两个星座/占星页面已需要的算法数据接入后台，让团队能查看、测试和追踪。

需要支持：

1. 查看出生星盘事实数据。
2. 查看每日星象或星座运势相关数据。
3. 查看某个模块使用了哪些算法字段。
4. 查看哪些算法字段进入了 Prompt。
5. 查看哪些算法字段直接进入最终 JSON。
6. 测试用户资料变化后，算法数据是否变化。

算法来源可以包括：

1. 现有后端接口。
2. 程序员自研算法。
3. 开源占星算法库。
4. 占星老师提供的规则和映射方法。

第一期只要求可查看、可追踪、可接入，不要求做完整算法编排系统。

## 13. 安全与知识产权保护

本后台会集中管理 Nexa 的核心资产，包括算法、Prompt、知识库、字段契约、模型配置、测试样本和调用日志。

这些内容涉及产品竞争力和知识产权，必须在第一期就考虑安全保护。

需要重点保护：

1. 占星算法规则和算法输出。
2. 占星老师提供的规则、权重、映射方法。
3. Prompt 模板、系统提示词、模块提示词。
4. Markdown 知识库和 RAG 知识片段。
5. 模型 API Key、供应商配置、调用参数。
6. 用户出生资料、个性化偏好、历史生成内容。
7. 成本数据、调用日志、灰度策略。
8. 前端 JSON 字段契约和产品内容样例。

第一期必须支持：

1. 账号登录。
2. 基础角色权限。
3. 敏感信息脱敏。
4. API Key 不明文展示。
5. 关键操作审计。
6. 接口鉴权。
7. HTTPS 或等效安全访问。
8. 必要时限制 IP、VPN 或内部网络访问。

目标是防止外部人员通过网站、接口或日志把 Nexa 的算法、知识库、Prompt、API Key和模型配置全部扒走。

调用大模型时也要遵守最小必要原则：

1. 只发送当前模块生成所需的内容。
2. 不把完整知识库整库发送给模型。
3. 不把完整算法规则暴露给模型。
4. 用户隐私信息尽量最小化。
5. 记录不同模型供应商的数据使用风险。

## 14. 问题排查流程

当产品经理发现页面内容有问题时，需要能在后台完成以下动作：

1. 选择出问题的页面。
2. 选择出问题的模块。
3. 查看该模块当时使用的算法数据。
4. 查看该模块当时发送给大模型的完整请求内容。
5. 查看大模型原始返回内容。
6. 查看最终返回给前端的 JSON 字段。
7. 查看该模块使用的 Prompt 版本、模型版本、知识库内容和负责人。
8. 标记问题类型。
9. 指派负责人处理。
10. 处理后重新测试。
11. 对比修复前后的输出。
12. 测试通过后发布或灰度。

问题类型至少包括：

1. Prompt 问题。
2. 算法数据问题。
3. 接口字段问题。
4. JSON 拼装问题。
5. 知识库问题。
6. 模型输出问题。
7. 前端展示问题。
8. 产品样例或字段定义问题。

## 15. 新增模块流程

平台第一期必须支持快速新增模块。

新增模块不是只创建一个模块名称，而是要能完整进入后台管理、测试和发布链路。

新增模块流程：

1. 在模块中心点击新增模块。
2. 选择所属页面。
3. 输入模块名称。
4. 选择是否从已有模块复制。
5. 配置输出 JSON 字段契约、字段说明和内容样例。
6. 上传或填写系统提示词和用户提示词。
7. 选择需要使用的算法数据。
8. 选择需要使用的知识库。
9. 选择默认大模型。
10. 配置 Fallback 内容。
11. 保存为草稿。
12. 自动出现在模块中心。
13. 可以进入测试中心测试。
14. 测试通过后提交审批。
15. 审批后灰度发布。
16. 灰度通过后正式上线。
17. 如果上线后有问题，可以单独回滚或停用。

新增模块的目标是：

1. 不需要重新开发整页。
2. 不影响其他模块。
3. 可以马上生成新的 API 输出内容。
4. 可以快速接入前端需要的新 JSON 字段。
5. 可以让产品、Prompt、程序员围绕新模块直接协作。

## 16. Fallback 规则

每个模块必须有 Fallback 内容。

当模块生成失败时：

1. 不影响其他模块。
2. 不影响整页返回。
3. 当前模块返回 Fallback 内容。
4. 后台记录失败原因。
5. 负责人可以在模块中心看到异常。

Fallback 可以由固定文案、算法基础内容或备用模型生成，具体实现由程序员根据成本和稳定性决定。

### 16.1 Fallback 告警

只要触发 Fallback，后台必须产生告警记录。

告警需要显示：

1. 触发时间。
2. 触发页面。
3. 触发模块。
4. 影响用户或测试样本。
5. 原本调用的模型。
6. 是否使用备用模型。
7. 最终返回了哪一种 Fallback。
8. 当前负责人。
9. 处理状态。

### 16.2 Fallback 触发原因

后台必须能看到为什么触发 Fallback。

原因至少包括：

1. 大模型 API 没有返回。
2. 大模型 API 超时。
3. 大模型 API 报错。
4. 大模型 API 费用、额度或余额超额。
5. 模型返回内容不是合法 JSON。
6. 模型返回字段缺失。
7. 模型返回内容触发安全规则。
8. Prompt 拼装失败。
9. 算法数据缺失。
10. 知识库检索失败。
11. 后端接口异常。
12. 未知原因。

每次触发 Fallback，都需要保留当时的请求内容、错误信息、模型返回内容和最终返回给前端的 Fallback 内容。

### 16.3 Fallback 预警

后台需要支持 Fallback 预警。

至少需要支持：

1. 某个模块 Fallback 次数异常升高时预警。
2. 某个模型连续触发 Fallback 时预警。
3. 因 API 费用、额度或余额问题触发 Fallback 时立即预警。
4. 因 JSON 格式错误连续触发 Fallback 时预警。
5. 因算法数据缺失触发 Fallback 时预警。

预警目标是让团队尽早发现问题，而不是等产品经理或用户看到异常内容后才知道。

## 17. 第一阶段交付目标

第一阶段完成后，团队应该可以做到：

1. 在后台看到出生星盘解读页和每日星座运势页的全部模块。
2. 为每个模块单独修改 Prompt。
3. 为每个模块选择不同大模型。
4. 上传 Markdown 知识文档。
5. 查看模块使用的算法数据。
6. 配置每个模块的输出 JSON 字段契约。
7. 单独测试某个模块。
8. 批量测试多个用户样本。
9. 查看大模型API调用总成本、当日成本、近 7 天成本、本月成本，以及每个模块的成本和调用情况。
10. 将模块版本提交审批。
11. 对模块进行灰度发布。
12. 出问题后单独回滚某个模块。
13. 把现有页面、现有 Prompt、现有 API 字段和现有算法输出接入后台管理。
14. 根据一次真实页面问题，查看当时的算法数据、大模型请求、大模型返回和最终 JSON。
15. 给问题标记类型并指派负责人。
16. 记录问题处理状态，直到测试通过或发布上线。
17. 对不影响前端结构的内容调整，可以直接通过后台配置、Prompt 或知识库进入测试流程。
18. 减少产品、前端、后端之间围绕字段样本反复沟通造成的等待。
19. 打通网页 APP 后端和大模型 API 后端，让一次页面请求可以被统一追踪、统一测试、统一排查。
20. 完成基础账号权限、敏感信息脱敏、关键操作审计和 API Key 保护。
21. 查看 Fallback 触发次数、触发原因和告警记录，并能识别是否由 API 费用、额度、模型无返回、JSON 错误或算法数据缺失导致。
22. 可以在后台新增模块，并让新模块进入字段配置、Prompt 配置、模型选择、测试、灰度、发布和回滚流程。

## 18. 一期不做的内容

以下内容不放入第一期：

1. 八字页面接入。
2. 完整八字后台迁移。
3. 自研模型训练。
4. 复杂模型训练平台。
5. 多占卜体系全部接入。
6. 复杂 CRM 或用户运营系统。
7. 复杂组织权限体系。
8. 过度复杂的数据分析大屏。

第一期不做复杂组织权限体系，但必须具备基础账号权限、角色限制、敏感信息保护和关键操作审计。

## 19. 后续接入策略

第一期星座/占星两页跑通后，再评估接入八字。

后续路径：

1. 星座两页验证后台是否真实可用。
2. 团队确认模块管理、Prompt 调试、字段契约、测试、发布、回滚、成本统计流程是否顺畅。
3. 确认两个后端打通方式稳定。
4. 确认安全和权限方案可接受。
5. 再逐步把八字模块、八字提示词、八字算法数据和八字页面成果接入后台。

八字在此之前继续按照原开发模式推进，不因后台一期验证而中断。

## 20. 总结

第一期最重要的目标不是功能完整，而是可用。

一周内需要做出一个能真实帮助团队协作的版本。

这个版本要解决：

1. 星座两页内容怎么生成。
2. 模块出了问题怎么定位。
3. Prompt 怎么快速调整。
4. JSON 字段样例怎么统一管理。
5. 前端、后端、产品、Prompt 调试人员怎么减少来回沟通。
6. 两个后端怎么打通。
7. 成本和模型怎么管理。
8. 安全和知识产权怎么保护。

系统稳定后，再把八字成果接入进来。
